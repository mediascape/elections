MEDIASCAPE'S ADAPTATION TOOLKIT API

This document describes the libraries developed in MediaScape to address the multi-device adaptation challenge in work package 5. Following the MediaScape architecture, work package 5 provides three different libraries that can be used independently but that work as a main umbrella library to deal with the adaptation challenge in multi-device media applications. This document technically details the libraries, providing the API documentation, examples, licenses and a link to the code in the repository.

  • Multi-device adaptation 

  • User interface adaptation 

  • Personal and shared device adaptation 

  • Inter-components communication 

The first two adaptation challenges are considered to be of higher priority, and are directly addressed by the Multi-device Adaptation Engine library and the User Interface Engine library. The personal and shared device adaptation challenge is an important particular case of Multi-device Adaptation Engine. Finally, the inter-components communication is enabled through the Adaptation Toolkit library, providing the developer with a uniform way to use the adaptation libraries in an application based on Web Components.

Multi-device Adaptation

The Multi-device Adaptation Engine aims to dynamically decide which part of application content goes to which device, according to the criteria defined by the application developer, the current context and device capabilities. Within the MediaScape project, application content has been abstracted as a set of web components that can be distributed over multiple devices within a single application session. Meanwhile, the capabilities of all active devices have been shared across agents via the shared application context, provided by the interface of the ApplicationContext module in WP4. Therefore, the problem for Multi-device Adaptation Engine can be formulated as follows: given a set of web components and a set of active and changeable devices with their capabilities, the Multi-device Adaptation Engine needs to figure out how to allocate the web components to the active devices in a proper way so that the distribution of web components over devices can fit in the current context and be able to enable seamless user experience.

The Multi-device Adaptation Engine takes into account three main requirements: 1) explicit adaptation behaviour, meaning that for a specific context application developers know how to allocate each web component to each device; 2) implicit adaptation behaviour, meaning that application developers know which type of device is required or preferred by each specific web component but do not have to define specific context; 3) user-preference-based adaptation behaviour, users can use intuitive interfaces to move or copy web components from one device to another device according to their own preferences on top of the default behaviour provided by the application. As illustrated by Figure a plugin based framework has been implemented to realise the Multi-device Adaptation Engine. Different adaptation behaviours are supported by different adaptation plugins and all of them provide decisions to the Multi-device Adaptation Engine with a predefined priority between behaviours. In the end, it is up to the Multi-device Adaptation Engine to make the final combined decision and then to filter out the required results for the User Interface Engine.

  • allow application developers to define different rules to specify different adaptation behaviours for their multi-device web applications; 

  • support three major adaptation behaviours 

    • explicit adaptation behaviour: CA rules,using a set of condition-action rules to define how web components should be distributed under a known context  

      • implicit adaptation behaviour: due to the dynamic nature of the multi-device context of an end user, it is not affordable to define an explicit adaptation behaviour for each one of the conditions. The implicit adaptation behaviour provides an implicit way to add behaviour hints, preferences and requirements for the application and each one of the web components, then to decide at run-time how to adapt the components to a specific set of devices. At the moment, the Multi-device Adaptation Engine provides two specific implicit adaptation behaviours, but the API enables the creation of different behaviours using specific properties. 

        • bestfit: define the preferred criteria for specified web components to find the best device, such as the device with the largest screen size, the device with the best network connection, the most interactive device, etc. 

        • affinitymatching: define requirements and preferences for each individual web component in terms of their required and preferred device capabilities; for each specific web component, its requirements are used to find out the possible destination devices that fit the required device capabilities and then its preferences are used to rank them based on their affinity scores; in the end, an optimization algorithm is used to automatically allocate each web component to the best destination device by maximizing the global affinity score. 

      • user preference based adaptation behaviour: on top of the default adaptation behaviour offered by the Multi-device Adaptation Engine, the application developer can also define the personalisation level of the end user, offering them pop-up interaction dialogues to choose which web component to duplicate or move from one device to another. 

    • use the priority of each adaptation plugin to solve/control the potential conflicts among different adaptation behaviours 

    • support JSON based rule definition and make it configurable to application developers 

    • allow advanced application developers to implement their own adaptation plugins via a unified plugin interface 

    API definition

    • it accesses the latest context (e.g., how many devices in the current application sessions and the capabilities of each active device) from the interface of the shared applicationContext module that has been developed in Work Package 4; 

    • it provides an interface for application developers to specify the adaptation behaviour. It is a JSON-based rule definition defining the explicit adaptation Condition-Action rules, and the custom properties for the implicit adaptation; 

    • It also provides a unified interface for advanced application developers to implement new adaptation plugins for extending the capability of the adaptation engine. For example, to add a new implicit rule plugin based on specific implicit properties. 

    Interface for developers to specify the adaptation behaviour of the application

    A JSON-based rules file is used by web developers to specify the adaptation behaviour for their applications and this rule file must be placed into the root folder of the applications.

    JSON is selected to define adaptation rules since it is well known to web developers and also both simple and flexible to express different types of rules. More important, JSON is supported by all browsers natively and no additional library is needed to parse the rules file.

            {
            "name": "Multi-device adaptation behaviour descriptor for the multi-device adaptation engine",
            "explicitRules":  {rule object},
            "implicitRules": {
              "bestfit": {rule object},
              "affinitymatch": {rule object}
            },
            "userPreferences": {rule object}
        }

    Each rule object includes the following five types of information:

    • name: a unique name for the plugin; 

    • enabled: tell the adaptation engine to enable it or disable it; 

    • capability: to explicitly specify what are the agent capabilities needed for this adaptation plugin; this is for the adaptation engine to know what are the agent capabilities to subscribe; 

    • priority: the priority to be considered by the Multi-device Adaptation Engine when merging decisions from multiple adaptation plugins; 

    • behaviour: how to adapt, to specify the detailed adaptation behaviour 

    The Multi-device Adaptation Engine must be initialised with the location of the rules file. Since the Multi-device Adaptation Engine is able to work together with the Adaptation Toolkit library, the Adaptation Toolkit library must be included first by the application. Then the application must call the init function to specify where the rules file is located (see example below). Meanwhile, the shared application context will be created and be set as an input parameter for the multi-device adaptation engine. 

          // initialize the adaptation engine
      var rulefile = 'rules.json';
      var AE = mediascape.AdaptationToolkit.Adaptation.multiDeviceAdaptation;
      AE.init(rulefile, appContext);
      // appContext is the shared application context, which must be created first by the application
      // which is a tasks the adaptation toolkit.

    Interfaces for developers to create new plugins inside Multi-device Adaptation Engine

    New plugins can be added to support new rules. Each one of the plugins needs to follow a specific skeleton or structure, so the Multi-device Adaptation Engine can manage and prioritise the decisions across the different behaviours and plugins.

    Plugins must implement an init function. This function is trigerred by the Multi-device Adaptation Engine during the initialisation phase. Plugins must also expose an onChange function, called by the Multi-device Adaptation Engine whenever a new context change happens, such as a new device joining the session. The following code provides an example of the structure of a plugin:

        //init function, triggered by the multi-device adaptation engine during the initialization phrase
    this.init = function(rules, ctx);
  /** rules represent the specification of adaptation behaviours, which will be defined by application developer
      ctx is the initial shared application context, given by the multi-device adaptation engine
      onChange function, triggered by the multi-device adaptation engine whenever a new context change happens */
    this.onChange = function(evt, ctx);
 /** evt represents the new change event to the shared application context
     ctx represents the latest application context, given by the multi-device adaptation engine */

    Once the new adaptation plugin is implemented, it must be placed into the plugins folder of the adaptationtoolkit library. See in Section 3.3 the Adaptation Toolkit library. The Multi-device Adaptation Engine is a library that can be used on its own, but the Adaptation Toolkit library aims to homogenise and to provide a way to use the Multi-device Adaptation Engine and the User Interface Engine together with all the potential of the adaptation tools with UIEngine.

    Examples

    A Hello World example has been provided with the API and the documentation. The example supports three different behaviours: explicit rules, implicit rules and user preferences. They provide a different perspective when defining the multi-device behaviour of the application. For each of them, an example is provided to show how they should be defined and how the Multi-device Adaptation Engine behaves consequently. The Hello World example assumes that there are three web components (chart, view and map) in the application.

    • explicit rules 

    For the specification of the explicit adaptation behaviour, the "agents" part lists all possible agents with different capabilities and then the "rules" part lists a set of specific conditions and their associated actions to do under those conditions. For the actions, we have to explicitly define which agent is going to show which components. This way, the developer can explicitly define how to allocate the web components across the different devices for a specific context. For instance, if there are two devices, one of them has a screen size bigger than a value (let's say a TV or a laptop) and the other has a small screen size and it is touchable (let's say a smartphone or a tablet), show A, B, and D components in the big device and B and C components in the small device. The following code provides how to express these explicit Condition-Actions on the JSON file. 

    {
"explicitRules":{
 "name":"explicit",
 "enabled":true,
 "priority":2,
 "capabilities":[
    "deviceType"
 ],
 "behaviour":{
    "agents":{
       "agent1":{
          "deviceType":"computer"
       },
       "agent2":{
          "deviceType":"tablet"
       },
       "agent3":{
          "deviceType":"mobile"
       },
       "agent4":{
          "deviceType":"tv"
       }
    },
    "rules":[
       {
          "conditions":{
             "exists":[
                "agent1"
             ]
          },
          "actions":{
             "agent1":{
                "chart":{
                   "load":true
                },
                "view":{
                   "load":false,
                   "volume":1
                },
                "map":{
                   "load":false
                }
             }
          }
       }

    ]
 }
}

    • Implicit rules 

    However it is not affordable to cover explicitly all the possibilities that could happen with an application of N components and a user using 1, 2, 3 or M heterogeneous devices at the same time. This is why implicit rules should be defined. The following example shows the affinitymatch rule, where the system provides the most appropriate device for each one of the components, based on defined properties for each component. In the behaviour specification, the developer needs to define the required and preferred device capabilities for each component. The adaptation engine will calculate the best match automatically. In the preference part, an affinity score is defined for each preferred device capability. 

        "affinitymatch": {
            "name": "affinitymatch",
            "enabled": false,
            "priority": 4,
            "capabilities": ["platform","onLine"],
            "behaviour": [
                {
                    "componentId": "chart",
                    "requirement": "platform(Win32)",
                    "preference": "onLine(high, 2)"
                },
                {
                    "componentId": "view",
                    "requirement": "platform(linux)",
                    "preference": "onLine(high, 1)"
                },
                {
                    "componentId": "map",
                    "requirement": "platform(android)",
                    "preference": "onLine(high, 1)"
                }
            ]
 }

    • User preference rules 

    For user preference based adaptation, the developers just have to define which components can be duplicated or moved from one device to another device in the rules file. The rest is done by users via a pop-up dialogue, interacting with the default presentation of the application. 

          "userPreferences": {
        "name": "userpref",
        "enabled": true,
        "priority": 1,
        "capabilities": ["deviceType", "operation"],
        "behaviour": [
            {
                "componentId": "chart",
                "movable": true,
                "duplicable": false
            },
            {
                "componentId": "view",
                "movable": true,
                "duplicable": false
            },
            {
                "componentId": "map",
                "movable": true,
                "duplicable": true
            }
        ]
 }

    User Interface Engine

    MediaScape targets applications that provide shared experiences across multiple and highly heterogeneous devices. The User Interface Engine provides mechanisms to create responsive and adaptive user interface layouts for those media applications. When application developers are facing a single-device user interface, they define CSS templates to organise the items in the layout, usually creating a different template for each target device through Media Queries. However, when application developers are dealing with multi-device applications, this approach becomes problematic. For instance, for an application with 6 different items, within a multi-device scenario, a single device could show all the components, only 5 of them, 4 of them, etc. Furthermore, when for example 4 items are shown on a device, the combinatorial possibilities of selecting 4 components from a total of 6 yields a total of 15 different options. As a result, an application of 6 items has 64 different combinations to create a layout template for each target device. This library provides a more versatile solution in order to create responsive and adaptive User Interfaces depending on the multi-device context of the user, based on layout templates that organise the elements in the layout on a dynamic way through semi-automatic patterns and behaviour.

    Multi-device Adaptation Engine manages the multi-device behaviour of the application depending the number of devices that the final user is using simultaneously and the multi-device rules that the application developer defines. Thus, the User Interface Engine dynamically creates a responsive layout organising the output components provided by the Multi-device Adaptation Engine and applying rules according to the UI properties that the application developer defines for each component and for the application itself. These rules take into account parameters from the context information such as features of the device (screen size, capabilities, etc.) and decide, based on templates, the spatial scheme of the components. The layout is generated over CSS properties including media queries to integrate responsive design mechanisms.

    API definition

    The User Interface Engine library provides two different running modes: Production and Development. The Production mode consists of a minified JavaScript library that provides a quick deployment possibility with all the needs of the library to run it over a Web Server (e.g. Apache server) and start using it immediately. The Development mode provides a RequireJS-based11 modularised JavaScript library to explore the code, debug the library and development issues.

    The library is optimised for Google Chrome Web browser and some of the layouts require to active the Web Platform flag in chrome://flags/.

    The User Interface Engine library has to be included in the application:

            <script src="js/mediascape.uiengine.min.js"></script>

    The application has to be developed following the Web Components specification12. The different elements to adapt in a multi-device context should be defined as Web Components. The User Interface Engine module must be activated and the behaviour parameter set through the init function:

    init(components,LAYOUTMODE)

    This command activates and initialises the main User Interface module. The components parameter defines the components to be displayed and organised through the User Interfaces across the different devices. Typically, they will be all the Web Components defined in the application. The LAYOUTMODE parameter sets the behaviour of the User Interface library, allowing now three different working modes:

  • STATIC: The application developer chooses a User Interface layout template (e.g. the Picture-in-Picture layout template). This template will be used under all the circumstances. The selected layout has to be defined in the 'layouts' folder, and has to follow a specific skeleton (see below the Layout Constructor details, to understand the layout template structure and to create a new layout template). 

  • CUSTOM: The application developer chooses a User Interface layout to be applied under all the circumstances, like in the STATIC mode. However, it provides also a set of other layout templates in the 'layouts' folder. This mode provides the end user the option to interact with the application and change the layout template to one of the proposed templates by the application provider. 

  • ADAPTABLE: The User Interface Engine library decides the most appropriate layout template depending on circumstances described by the application developer in the 'layouts' folder. The decision can be based on the context parameters, such as the number of components to be shown, the screen size of the device, the type of user interaction supported by the device, etc. (see below the findBestLayout() function to create the template selection rule depending the context information). Moreover, apart from providing the appropriate default layout template under any circumstance, this mode provides also the end user with the option to change the layout template at run-time to one of the available templates. 

    The following code provides an example that initialises the User Interface Engine library to ADAPTABLE mode.

    document.addEventListener('adaptationToolkit-ready',function(data){
    var UI = mediascape.AdaptationToolkit.Adaptation.UIAdaptation;
    UI.init(data.detail.components,UI.LAYOUTMODE.ADAPTABLE);
},false);
    The following code provides an example that initialises the User Interface Engine library to the STATIC mode using the Picture-in-Picture (PiP) layout template. As shown in the example, the useLayout function is also needed for the STATIC and CUSTOM modes to select the default template in all circumstances: 

    document.addEventListener('adaptationToolkit-ready',function(data){
  var UI = mediascape.AdaptationToolkit.Adaptation.UIAdaptation;
  UI.init(data.detail.components,UI.LAYOUTMODE.STATIC);
  UI.useLayout(pip);
},false);

    useLayout(layoutName)

    Use this function to define the default layout template to be used by the library; it is mandatory in the STATIC and CUSTOM modes. Otherwise, the library will select the default template from one of the available templates arbitrarily:

      UI.useLayout(pip);

    At this point, the User Interface Engine library is ready to work and will provide a responsive User Interface on each device, showing the different components of the application across the different devices. These are all the steps that need to be followed if an application developer or a broadcaster wants to use the User Interface Engine library and wants to re-use the layout templates already created and available (see below for examples of layout templates that have been created during the project). However, the User Interface Engine library aims to be flexible and modular. For this reason, the library is agnostic of the layout template, and allows the insertion of new layout templates. Thus, the API provides also the structural information needed to create custom layout templates.

    With this purpose, the User Interface Engine library provides a clear structure of functions that each layout template needs to create. This way, on the one hand the library decides which layout to load depending the circumstances, and re-directs to the specific layout all the events that might happen during the run-time. The API provides a layoutConstructor to create a new template. A layout template must follow the following interface (see below for concrete code examples):

    render()

    It is the main function that the layout template should define, describing the organisation itself of the elements across the display.

      render = function (cmps){
      console.log("define the arrangement of the components in the layout");
   }

    onComponentsChange()

    The layout template should define the actions when there is a change on the list of components to be shown on a device. For instance, if a TV is showing the components A, B and D and there is an event fired by the Multi-device Adaptation Engine updating the list of components to show to A, B, C and D, the User Interface Engine library will run the onComponentsChange function of the appropriate layout template. 

      onComponentsChange = function (cmps,cmds){
      console.log("Adaptation engine has decided to show/unshow some components, is it necessary to change layout setting");
  }

    onOrientationChange()

    The layout template should define the action when there is an orientation change on the device. For instance, when a tablet goes from landscape to portrait. Typically, this will call the render function of the same layout template or, depending on the circumstances, a new layout template will be chosen. 

      onOrientationChange = function (cmps){
    console.log("Orientation of the device has changed, is it necessary to change layout setting ?");
  }

    onLayoutChangeEvent()

    The layout template should define the action to be performed when the end users wants to change the layout template proposed by default by the library. Typically, the user interacts through a button and this function calls the findBestLayout function (see below) to calculate the next most appropriate layout from the available ones. 

      onLayoutChangeEvent = function (cmps){
    console.log("what to do when the user/system has changed the layout");
 }

    onResizeEvent()

    The layout template should define what to do when the Browser size changes to provide a responsive User Interface. Typically, it will adjust the render parameters to the new context situation. 

      onResizeEvent = function (cmps){
       console.log("what to do when the size of the Browser changes");
   }

    onUnload()

    The layout template should define what to do to unload all the modifications done by it. Typically it will be used when a new layout is going to be rendered. The layout templates may change the attributes of the components, such as the width or the height, and they also add interaction events (e.g. the menu layout provides a menu to select the component to be shown in full screen). The onLoad function removes all the modification introduced by the layout template. 

      onUnload = function (cmps){
         console.log("what to do to unload the layout ");
  }

    All the available layout templates have to be in the 'layouts' folder in the mediascape/AdaptationToolkit/Adaptation/UIAdaptation/layouts/ path and declared in the JavaScript library file (mediascape/AdaptationToolkit/adaptation/UIAdaptation/UIAdaptation.js):

        define(["AdaptationToolkit/adaptation/UIAdaptation/layouts/layoutName1",
           "AdaptationToolkit/adaptation/UIAdaptation/layouts/layoutName2",
           "AdaptationToolkit/adaptation/UIAdaptation/layouts/layoutNameN"]

    The API provides a validation method for the created layout templates. It verifies the adequate registry of the layout in the library and ensures it fulfils all the necessary functions and definitions. It also provides debugging information with errors and warnings:

        console.log(layout.checkForImplementation());

    If it returns 0 errors, it means that the validation has been successful and that the layout is ready to be used.

    findBestLayout()

    The User Interface Engine library also provides a findBestLayout function that the application developer or broadcaster might want to change. This function decides the layout template to be used in the ADAPTABLE mode depending on the context information. The function returns an array of the list of the available layout templates ordered from the most appropriate one to the less appropriate one. 

          var UI = mediascape.AdaptationToolkit.AdaptationEngine.UIEngine;
      var bestL = UI.findBestLayout();
      console.log( bestL[0].name );
     /* e.g. returns 'menu' as the most appropriate layout*/
      console.log( bestL[1].name );
     /* e.g. returns 'pip' as the second most appropriate layout*/

    In order to facilitate the development under the User Interface Engine library, such as programming the findBestLayout function, the API provides a getLayouts function that returns all the registered layout templates in the system.

          var UI = mediascape.AdaptationToolkit.AdaptationEngine.UIEngine;
      var layouts = UI.getLayouts();
      /* e.g. returns list of layouts ['pip','menu']*/

    The User Interface Engine library is designed to organise the elements defined in the application following a layout template, depending on the context information and on some properties defined by the application developer. The User Interface Engine does not define a mandatory way to define those properties and it is something that could be addressed in different ways within each layout template. However, the Adaptation Toolkit provides a uniform way to parse CSS-like attributes in the layout templates (see Section 3.3 where the Adaptation Toolkit API is described and below where the properties used by the different layout templates are described).

    Layout Construction: Implementation

    These is the guideline to add a new layout to UIAdaptation module.

    The first thing is to create a new file with the .js extension in the following folder:

    mediascape/AdaptationToolkit/Adaptation/UIAdaptation/layouts/

    For example: menuLayout.js

    The next step is to add the following structure to the new file:

      define(["AdaptationToolkit/adaptation/UIAdaptation/layoutConstructor"],
    function(LayoutConstructor){

      var menuLayout = new LayoutConstructor('menuLayout');
      menuLayout.onComponentsChange = function (cmps){
          console.log("test");
      }
      menuLayout.render = function (cmps){
        console.log("test");
      }
      menuLayout.onOrientationChange = function (cmps){
        console.log("test");
      }
      menuLayout.onLayoutChangeEvent = function (cmps){
        console.log("layout changed");
      }
      menuLayout.onUload = function (cmps){
        console.log("unloading layout ");
      }
      menuLayout.__moduleName = menuLayout.name;
      return menuLayout;

    });

    And finally, add a new module parameter within UIAdaptation (mediascape/AdaptationToolkit/adaptation/UIAdaptation/UIAdaptation.js):

      define(
    ["AdaptationToolkit/adaptation/UIAdaptation/layouts/pip","AdaptationToolkit/adaptation/UIAdaptation/layouts/menuLayout"]

    Layout Construction: Validation

    The following codes verifies the adequate registry of the new layout:

      var layouts = mediascape.AdaptationToolkit.Adaptation.UIAdaptation.getLayouts());
  console.log(layouts[1].checkForImplementation());

    For a successful validation it will return 0 errors. It means the layout is ready to use.
